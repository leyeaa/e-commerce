# Experiment protocol

## Research question

For the selected Django REST API endpoints, how do human-authored deterministic
tests and Schemathesis schema-driven fuzzing differ in defect detection,
coverage, setup effort, execution effort, and types of faults detected?

Exploratory testing is reported as an additional required method and is not
merged into the primary two-technique comparison.

## System and scope

- Baseline revision: see baseline.md.
- Database: PostgreSQL.
- Endpoint A: POST /api/cart/update/.
- Endpoint B: POST /api/orders/create/.
- Authentication: JWT bearer token.
- Fuzzing schema: OpenAPI 3.0.3 generated by drf-spectacular.

## Behavioural oracles

1. A request must not modify another user data.
2. Invalid input must produce a documented 4xx response rather than a 5xx.
3. A rejected request must not mutate persistent state.
4. Order creation must be all-or-nothing.
5. Order totals and items must equal the server-side cart state.
6. Empty carts must not produce orders.
7. Required checkout data must satisfy the API contract.
8. Successful responses must conform to the OpenAPI response schema.

## Independent execution order

1. Freeze this protocol, known-defect register, manual suite, fuzz settings, and
   exploratory charters.
2. Reset the PostgreSQL database to deterministic fixtures.
3. Run the human-authored suite and save its output and coverage.
4. Reset the standalone fuzz database.
5. Run Schemathesis with the recorded seed and request budget.
6. Conduct exploratory sessions against the same baseline revision.
7. Debrief and triage only after all independent runs are complete.
8. Deduplicate failures by root cause.
9. Fix confirmed defects in later commits and rerun all applicable checks.

## Primary measures

- Number of unique newly discovered confirmed defects.
- Defect category and severity.
- Detection overlap between techniques.
- Line and branch coverage.
- Test or request execution count.
- Setup time and test-design time in person-minutes.
- Execution duration.
- False positives and unreproducible failures.

## Secondary measures

- Number of known K defects independently reproduced.
- Minimal reproducer availability.
- Number of raw failures per unique root cause.
- Database-integrity violations.
- OpenAPI conformance violations.

## Fairness controls

- Use the same immutable application revision.
- Start each technique with equivalent deterministic data.
- Do not convert one technique findings into another technique tests until the
  baseline comparison finishes.
- Record tool versions, commands, seed, time, and configuration.
- Count unique confirmed defects rather than individual failing requests.
- Report schema inaccuracies and tool failures separately from application
  defects.

## Defect classification

- K: known during preliminary review.
- N: first identified during experimental execution.
- Source: exploratory, manual, Schemathesis, or multiple.
- Status: candidate, reproduced, confirmed, fixed, or rejected.
- Severity: critical, high, medium, or low with written justification.
