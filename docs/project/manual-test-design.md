# Human-authored test design

## Purpose

The manual technique in the primary comparison is a deterministic,
human-designed API test suite. The tests were derived from endpoint contracts,
input partitions, security boundaries, persistence invariants, and transaction
properties. They are automated with pytest only to make execution repeatable;
their inputs and oracles were selected by a person rather than generated by a
fuzzer.

## Frozen suite

The suite contains 20 collected cases across the two experiment endpoints.
Nine are expected control passes and eleven are strict expected failures tied
to defects known before formal execution. A strict expected failure becomes an
unexpected pass if application behaviour changes.

| Endpoint | Design area | Cases | Expected result |
| --- | --- | ---: | --- |
| Cart update | authentication, valid owner update, missing field, unknown item | 4 | pass |
| Cart update | cross-user access, malformed quantity, zero quantity, malformed item ID | 4 | strict expected failure |
| Order create | authentication, valid order, empty cart, invalid phone partitions | 5 | pass |
| Order create | missing required fields, unknown payment choice, missing or non-string phone, persistence, atomicity | 7 | strict expected failure |

## Input partitions and boundaries

- Authentication: absent bearer identity versus authenticated identity.
- Ownership: item owned by the caller versus item owned by another user.
- Item identifier: existing positive integer, unknown positive integer, and
  malformed non-integer value.
- Quantity: valid positive integer, missing, zero boundary, and structured
  non-integer value.
- Cart state: non-empty versus empty.
- Phone: valid 10 digits, alphabetic, below the 10-digit boundary, missing, and
  non-string structured data.
- Required checkout strings: present versus omitted name or address.
- Payment method: documented choice versus an unknown choice.
- Order persistence: successful cart-to-order mapping and injected failure
  during the second order-item write.

## Oracles

Each test checks more than the status code where relevant. Persistence checks
cover cart mutation, order count, line-item contents, totals, ownership,
checkout-field storage, and all-or-nothing transaction behaviour. The shared
behavioural oracles are defined in experiment-protocol.md.

## Contamination rule

The frozen manual cases are not to be changed after formal Schemathesis results
are inspected. New regression tests derived from formal fuzz or exploratory
findings belong in a later post-comparison suite and are reported separately.
